Multi-Region Configuration & Disaster Recovery

Automating environment consistency and multi-cloud resiliency with Ansible and Backblaze B2.

AnsibleMulti-CloudDisaster Recovery

01. Ephemeral Control Plane

Instead of a standing Ansible server, the GitHub Runner acts as a temporary node. This eliminates monthly EC2 costs and ensures every deployment starts from a 100% "known good" environment, preventing configuration drift on the management side.

02. Dynamic Inventory

I utilized AWS Dynamic Inventory and EC2 Tags to group servers. This removes the manual overhead of updating IP addresses; if an instance scales or changes, Ansible discovers it automatically. The CI/CD uses a matrix strategy to target multiple regions in parallel.

03. Multi-Cloud Backup Strategy

To avoid Single Cloud Failure, backups are streamed to Backblaze B2. For security and performance, backups are staged in a local temporary folder and immediately destroyed after the upload to minimize the data footprint and maintain a zero-trust environment.
