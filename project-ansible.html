Multi-Region Configuration & Multi-Cloud Recovery Framework

Ansible GitHub Actions AWS Backblaze B2 (DR) DevSecOps

The Mission

To provide consistent configuration management, security hardening, and database durability across a multi-region AWS footprint, ensuring zero dependency on a single cloud provider for critical backups.

Design Decisions

Decision: Decoupled Provisioning vs. Configuration

While many teams use Terraform for everything, I enforced a strict Separation of Responsibilities: Terraform manages the Infrastructure, while Ansible manages the State.
Why? This reduces coupling and allows the configuration lifecycle (e.g., patching, user management) to move faster than the underlying infrastructure lifecycle.

Decision: Multi-Cloud Resilience (AWS to Backblaze B2)

Database backups (MongoDB/Redis) are exported from AWS and stored in Backblaze B2.
Why? In a "Worst Case" banking scenario (account lockout or provider-wide outage), having a secondary cloud for recovery is a critical compliance requirement for Cyber Resilience.

Decision: Manual Approval Gates for Database Restore

I designed a manually triggered pipeline for database restores, requiring explicit operator input for the target host and backup identifier.
Why? Automated restores in production are dangerous. This "Human-in-the-loop" design ensures deterministic recovery and prevents accidental data overwrites.

Operational Architecture

The framework utilizes a dynamic AWS inventory to discover hosts in real-time across regions, ensuring that configuration management scales automatically as Terraform adds new compute resources.

Key Outcomes

  • Security Hardening: Automated S3 bucket policy enforcement (private ACLs, versioning) across all global resources.
  • Operational Agility: Single-command regional expansion—adding a region to the Ansible matrix requires zero logic changes.
  • Disaster Recovery: Proven cross-cloud restore workflow for MongoDB and Redis workloads.
