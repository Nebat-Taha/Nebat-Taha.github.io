Multi-Region Configuration & Multi-Cloud Disaster Recovery

Automating post-provisioning state and off-site resilience using ephemeral Ansible runners.

Ansible GitHub Actions Multi-Cloud Backblaze B2

01. Ephemeral Control Plane

Architectural Goal: Cost & Clean State

Instead of a standing Ansible server, the GitHub Runner acts as a temporary node. This eliminates monthly EC2 costs and ensures every deployment starts from a 100% "known good" environment with fresh dependencies, preventing configuration drift on the management side.

02. Dynamic Inventory & Multi-Region Matrix

Architectural Goal: Avoiding Static Overhead

I utilized AWS Dynamic Inventory and EC2 Tags to group servers. This removes the manual headache of updating IP addresses; if an instance scales or changes, Ansible discovers it automatically. The CI/CD uses a matrix strategy to target multiple regions in parallel.

03. Multi-Cloud Backup Strategy

Architectural Goal: Resiliency via Backblaze B2

To avoid "Single Cloud Failure," backups are streamed to Backblaze B2 using keys stored in GitHub Secrets. For security and performance, backups are staged in a local temporary folder and immediately destroyed after the upload to save disk space and minimize the data footprint.
