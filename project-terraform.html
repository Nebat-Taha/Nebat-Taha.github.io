Enterprise Terraform: Multi-Region Architecture & Governance

Architecting isolated Global vs. Regional lifecycles with S3 Backend Locking and Proactive Drift Detection.

Terraform AWS GITHUB ACTIONS S3 + DynamoDB Locking Matrix Strategy Governance Gates

01. Decoupled Resource Lifecycles

Infrastructure is split into two distinct CI/CD pipelines to ensure that changes to global identity (IAM) or DNS (Route53) do not interfere with regional compute resources.

Global vs. Regional Pipelines

Global Pipeline: Manages IAM and Route 53. Executed in us-east-1 as the authoritative region.
Regional Pipeline: Utilizes a GitHub Actions Matrix to concurrently deploy VPCs and EC2 instances across us-east-1, eu-central-1, and ap-southeast-2.

02. CI/CD Governance: Safety at Scale

Infrastructure as Code (IaC) requires more than just automation; it requires guardrails. I evolved the deployment pipeline into two distinct security models to handle production environments safely.

Continuous Deployment

Direct automation for dev/staging environments where rapid iteration and speed are prioritized.

Governance-First

Utilizes GitHub Environment protection and Artifact state-passing to ensure human verification before changes.

The Production Gate: Plan & Apply Split

By splitting the workflow into terraform_plan and terraform_apply, I introduced an Environment Gate. The workflow pauses after the plan is generated, requiring a manual review of the Terraform Artifact before the 'production' environment allows the execution.

Technical Implementation: Artifact Integrity

To ensure the exact code planned is the code applied, I implemented Artifact Passing. This prevents "state-drift" between the time the plan is generated and the time it is executed.


# Job 1: Saving the source-of-truth
- name: Upload Plan Artifact
  uses: actions/upload-artifact@v4
  with:
    name: tfplan-${{ matrix.region }}
    path: terraform/regions/${{ matrix.region }}/tfplan

# Job 2: The Production Gate
terraform_apply:
  needs: terraform_plan
  environment: production # Requires manual approval in GitHub UI

03. Collaborative State & Locking

Architectural Goal: S3 Backend with DynamoDB

To enable a collaborative team environment and prevent state corruption, I implemented a remote backend using Amazon S3 for state storage and DynamoDB for state locking. This ensures that concurrent deployment attempts are safely blocked, maintaining a single source of truth across all regions.

04. Proactive Drift Detection (Day 2 Ops)

Infrastructure "drift" occurs when manual changes are made via the AWS Console, breaking the link between code and reality.

Architectural Goal: Automated Integrity Checks

I engineered dedicated Drift Detection workflows using terraform plan -detailed-exitcode. The CI/CD uses logic gates to interpret exit codes: 0 (Safe), 1 (Error), or 2 (Drift Detected). If drift is found, the job fails and alerts the team for immediate remediation.

05. Shift-Left Security & Secret Management

Architectural Goal: Zero-Trust Deployments

For maximum security, AWS Access Keys are stored in GitHub Secrets and injected at runtime. Before any apply, the code must pass tfsec and Checkov static analysis gates to ensure all resources meet industry security standards.
