GitOps Continuous Delivery with ArgoCD and Minikube

This project demonstrates the implementation of a GitOps-based continuous delivery workflow using Minikube, ArgoCD, GitHub Actions, Docker, and Kustomize. The goal was to build an end-to-end deployment pipeline that follows GitOps principles, where Git repositories act as the source of truth for both application delivery and deployment state.

The implementation combines continuous integration, container image management, deployment automation, and declarative configuration management to create a repeatable deployment workflow. Application changes are automatically built, versioned, and promoted through Git-based workflows while ArgoCD continuously reconciles the desired state stored in Git with the running environment.

ArgoCD Docker GitHub Action Minikube GitHub

01.End-to-End GitOps Workflow

The project implements a complete GitOps workflow that automates application delivery from source code commit to deployment. Rather than deploying directly to the cluster, every change flows through a controlled process that uses Git repositories as the source of truth.

When code is committed to the application repository, GitHub Actions builds a Docker image, tags it (with the commit hash), and pushes it to Docker Hub. The pipeline then updates the deployment manifests stored in a separate GitOps repository. ArgoCD continuously monitors this repository and automatically reconciles the desired state with the running state inside Minikube, ensuring deployments remain consistent and traceable.

02. Dual-Repository GitOps Architecture

The project follows a common GitOps pattern by separating application code and deployment configuration into independent repositories. The application repository contains the source code, Dockerfile, and CI pipeline, while the manifest repository contains the Kubernetes deployment manifests that define the desired state of the environment.

This separation allows deployment configuration to be managed independently from application development activities. The manifest repository serves as the authoritative source of truth for deployment state, while ArgoCD continuously monitors that repository and applies changes to the environment. This approach improves visibility, simplifies change tracking, and aligns with established GitOps practices.

03. Cross-Repository Integration

A key component of the workflow is the automated integration between the application repository and the GitOps manifest repository. While each repository serves a different purpose, they must work together to enable continuous delivery.

GitHub Actions was configured with deploy keys and repository secrets that allow the pipeline to securely access the manifest repository. After building and publishing a new container image, the pipeline checks out the manifest repository, updates the deployment configuration, commits the change, and pushes it back to GitHub. This process allows deployment intent to be communicated through Git rather than direct interaction with the cluster, preserving the GitOps model throughout the deployment lifecycle.

04. Immutable Deployments with Git SHA Tagging

To improve deployment traceability, container images are tagged using the Git commit SHA rather than mutable tags such as latest. Each deployment therefore references a unique application version that can be traced directly to the source code commit that produced it.

Kustomize is used to manage image version updates within the manifest repository. During pipeline execution, the image reference is updated with the newly generated commit hash and committed back to Git. This approach creates immutable deployments, simplifies troubleshooting, and provides a clear audit trail of what version is running in the environment at any point in time.

05.Deployment Governance and Security Gates

The workflow incorporates approval mechanisms at both the continuous integration and continuous delivery stages, demonstrating how deployment automation can be combined with governance controls.

GitHub Environments were used to require approval before container images could be published to the registry, while branch protection and pull request requirements were applied to the GitOps repository before deployment-related changes could become part of the desired state. Together, these controls establish review points for both application promotion and deployment configuration changes.

06.Automated Platform Bootstrapping

To reduce manual setup effort and improve consistency, the project includes automation for provisioning the local GitOps environment. A dedicated bootstrap structure contains the resources and scripts required to initialize Minikube, install ArgoCD, create the required namespaces, and prepare the platform for application deployment.

The platform setup process is defined through reusable automation, allowing the environment to be provisioned in a repeatable manner with the same desired configuration each time. By codifying the initialization process, the project demonstrates Infrastructure as Code principles and establishes a consistent foundation for deployment and testing activities.
