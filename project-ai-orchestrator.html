AI-Driven Infrastructure

AI PR Orchestrator

Automating the bridge between Jira task management and GitHub version control using FastAPI and Local LLMs (Ollama).

Design Goal

The primary objective was to build a system that translates structured Jira tickets into functional code while maintaining strict architectural control, scalability, and clear system boundaries between project management and version control.

Extensibility

1. Label-Driven Execution

I utilized Jira labels to drive system logic. The orchestrator reads labels (e.g., terraform, python) to determine which prompt templates to load, which repository to target, and which specific CI workflows to trigger.

Impact: This removes hardcoded logic from the FastAPI application. Adding support for a new tool becomes a simple configuration change rather than a code deployment, allowing the system to scale with zero friction.
Separation of Concerns

2. Orchestrator as Control Layer

I introduced a lightweight **FastAPI** middleware to coordinate interactions. Jira defines the intent, the LLM handles the generation, and GitHub manages version control.

Impact: This creates a clear separation of concerns. The orchestrator is the only place where decision logic resides, simplifying maintenance and allowing each component of the stack to evolve independently.
DevOps Alignment

3. Prompt as Code

Prompts are stored as version-controlled .md files within the repository. They are treated as first-class citizens, identical to infrastructure or application code.

Impact: This makes AI behavior transparent and reviewable. Prompt changes must pass through standard Pull Request reviews, ensuring that the evolution of the AI's logic is governed by existing engineering standards.
Reliability

4. Structured Output Enforcement

The system enforces a strict JSON schema for all LLM responses. Instead of free-form text, the model must return data that the orchestrator can immediately parse and act upon.

Impact: This removes the inherent ambiguity of LLM outputs. It ensures downstream automation (like branch creation) can operate reliably without complex regex or secondary parsing logic.
Accuracy

5. Context Injection

The orchestrator dynamically injects live project context—such as existing module definitions and directory structures—into the prompt before it reaches **Ollama**.

Impact: By providing the model with real system information, we significantly reduce hallucinations. The generated code fits perfectly into the existing architecture rather than making generic assumptions.
Resilience

6. Retry with Error Context

If the GitHub CI/CD validation fails (e.g., a terraform plan error), the orchestrator captures the error logs and feeds them back to the LLM for a correction attempt.

Impact: This introduces an automated feedback loop. Instead of the process stopping at failure, the system attempts to self-correct, moving toward a more resilient, "self-healing" automation workflow.
Security

7. Scoped GitHub Bots

I implemented separate GitHub bot identities (via distinct Personal Access Tokens) for different domains, such as infra-bot for Terraform and app-bot for Python tasks.

Impact: This enforces the principle of **Least Privilege**. It also provides clear audit trails for automated changes, which is critical for security and compliance in a collaborative environment.
