AI-Driven Infrastructure Orchestrator

Automating the bridge between Jira task management and GitHub version control using FastAPI and Local LLMs (Ollama) to interpret labels and generate context-aware code.

PythonFastAPIOllamaJira API

01. Design Goal

Design a system that translates structured tickets into code, while maintaining control, scalability, and clear system boundaries. The goal was to ensure that while AI generates the code, the DevOps engineer remains the orchestrator of the logic and governance.

02. Label-Driven Execution

I used labels from Jira to determine which prompt to use, where the generated code should be placed, and which workflow should run. Instead of embedding logic for each technology, the system reads labels and maps them to configuration.

Impact: This removes hardcoded decision-making from the application and makes the system extensible. Supporting a new tool or workflow becomes a configuration change rather than a code change, which allows the system to scale with minimal friction.

03. Orchestrator as Control Layer

I introduced a lightweight orchestrator using FastAPI to sit between Jira, the LLM, and GitHub. This layer is responsible for handling the workflow and coordinating the interaction between systems.

Impact: This creates a clear separation of concerns. Jira defines intent, the LLM handles generation, and GitHub manages version control. The orchestrator becomes the only place where decisions are made, which simplifies maintenance and allows each component to evolve independently.

04. Prompt as Code

Prompts are stored as version-controlled .md files instead of being embedded in the application logic. They are treated the same way as infrastructure or configuration code.

Impact: This makes AI behavior visible and reviewable. Changes to prompts can go through pull requests, which aligns with existing DevOps practices and introduces control over how the system evolves.

05. Structured Output Enforcement

I enforced a strict JSON schema for all LLM responses instead of accepting free-form output. The system expects structured data that can be directly consumed.

Impact: This removes ambiguity from the LLM output and makes it reliable for automation. The trade-off is reduced flexibility, but it ensures consistency and allows downstream processes to operate without additional parsing logic.

06. Context Injection

The orchestrator dynamically injects relevant project context, such as module definitions and structure, into the prompt before sending it to the LLM. This ensures the model works with real system information.

Impact: This reduces hallucination and improves alignment with the existing codebase. The LLM generates output that fits within the actual system rather than making assumptions about it.

07. Retry with Error Context

When CI/CD validation fails, the system captures the error output and feeds it back into the LLM as additional context for a correction attempt.

Impact: This introduces an iterative feedback loop that reduces manual debugging. Instead of stopping at failure, the system attempts to correct itself, moving toward a more resilient workflow.

08. Scoped GitHub Bots

I used separate bot identities in GitHub for different domains, such as infrastructure and configuration.

Impact: This enforces least-privilege access and improves security. It also makes ownership of automated changes clearer, which is important in collaborative environments.
