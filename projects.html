Multi-Region Infrastructure Framework with GitHub Actions

Terraform GitHub Actions AWS S3/DynamoDB (State)

Managed a scalable multi-region AWS architecture using a modular Terraform structure and automated CI/CD pipelines. Designed to support rapid expansion into new regions with zero changes to the underlying deployment logic.

Design Decisions

Decision: GitHub Actions Matrix Strategy for Regional Deployment

Instead of sequential deployments, I implemented a strategy: matrix in GitHub Actions.
Why? This allows for parallel execution across regions (ap-southeast-2, eu-central-1, etc.). It ensures consistency while significantly reducing total build time as the regional footprint grows.

Decision: Decoupled Global vs. Regional State

I split the infrastructure into terraform-mr/global and terraform-mr/regions/.
Why? Banking-grade stability requires minimizing the "blast radius." Changes to regional compute shouldn't risk the global IAM or Networking backbone. Each region operates on its own independent state file with S3 versioning and DynamoDB locking.

Decision: Separate Drift Detection Pipeline

Created a standalone workflow for drift detection, separate from the main deployment pipeline.
Why? This enforces a clear separation between Change Execution and Change Detection. It allows for continuous monitoring of infrastructure health without the risk of an accidental 'apply' during a detection run.

Infrastructure Architecture

The project follows a "Golden Path" for regional expansion. Adding a region requires only two steps: creating a new folder and updating the matrix list.

Key Outcomes

  • Scalability: Successfully automated parallel deployments across 3+ AWS regions.
  • Security: Implemented encrypted remote state with strictly scoped IAM policies and MFA protection.
  • Maintainability: Reduced code duplication by 70% through a centralized /modules library consumed by all regions.
