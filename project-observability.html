Git-Driven Observability Platform

A robust observability ecosystem where infrastructure, configuration, and monitoring logic are entirely driven from Git to enable rapid recovery and zero manual intervention.

PrometheusGrafanaAnsibleTerraformAWS

01. Design Goal

The goal was to design an observability system where every component—from the underlying cloud resources to the alerting rules—is defined in code. This ensures consistent deployment across environments and enables the team to rebuild the entire monitoring stack from scratch in minutes if necessary.

02. Git as the Source of Truth

I designed the system so that infrastructure provisioning, configuration, and monitoring logic all originate from Git. Changes to Terraform modules, Ansible playbooks, or custom Python monitoring scripts trigger their respective CI/CD workflows automatically.

Impact: This removes manual configuration from the system and ensures total consistency. It makes the entire platform reproducible, allowing for reliable environment cloning and disaster recovery.

03. Infrastructure as an Entry Point

Terraform serves as the foundational layer, ensuring infrastructure is provisioned before configuration or monitoring logic is applied. The setup utilizes remote state management in S3 with DynamoDB locking to prevent concurrent state modifications.

Impact: This enforces a predictable provisioning order and prevents configuration drift. It allows the entire platform to be created or destroyed on demand, directly supporting cost optimization and testing scenarios.

04. OIDC-Based Authentication

Instead of relying on static, high-risk IAM credentials, I implemented OpenID Connect (OIDC) between GitHub Actions and AWS. Workflows assume IAM roles dynamically at runtime based on the repository's identity.

Impact: This eliminates long-lived secrets and drastically improves the security posture. Access is temporary, short-lived, and scoped specifically to each workflow execution.

05. Keyless Access with EC2 Instance Connect

I eliminated the need for manual SSH key pair management by leveraging EC2 Instance Connect. The GitHub runner generates a one-time temporary key and injects it into the instance metadata at runtime.

Impact: This removes the operational burden of secret rotation and secret management for SSH. Access becomes identity-based rather than file-based, making the system easier to share and more secure.

06. Separation of Provisioning and Deployment

Ansible workflows are split into two layers: a base "provisioning" workflow for system hardening and setup, and a lightweight "deployment" workflow for updating specific monitoring scripts.

Impact: This dramatically reduces execution time for frequent changes. Updating a single Prometheus exporter doesn't require a full system reconfiguration, leading to more efficient pipelines.
