Modernizing AWS Observability: From Silos to GitOps

Prometheus Grafana Python AWS SQS GitOps

In complex AWS environments, visibility is often trapped behind high-cost CloudWatch dashboards or siloed accounts. I engineered a centralized observability platform that monitors AWS SQS and infrastructure health using a 100% "Monitoring-as-Code" approach.

01. The Technical Engine

The solution utilizes Prometheus as the time-series core, pulling data from three distinct sources to create a unified view.

Custom Python SQS Exporter

Standard exporters often lack the granularity needed for specific queue behaviors. I developed a Python script using Boto3 that dynamically discovers all SQS queues and exposes real-time metrics (Messages Visible, Age of Oldest Message) on a Prometheus-compatible endpoint.

02. Evolving to a Collaborative Model

The most critical phase of this project was moving away from a "single-server" setup to a Collaborative GitOps workflow. This ensured the system was production-ready and team-accessible.

Feature Traditional Setup My Collaborative Model
Config Management Manual SSH edits Git-Controlled (Versioned)
Auth / Security Hardcoded IAM Keys IAM Instance Profiles (Roles)
Uptime Manual script execution Systemd Daemons (Auto-Restart)
Dependency System-wide Python Isolated Virtual Environments

03. Visualization & Visualization

Data is useless without clarity. I built Grafana Dashboards that use "Join by Field" transformations to correlate SQS throughput with infrastructure health metrics, providing a single pane of glass for SRE teams.

Key Outcomes

  • Security Hardened: Eliminated static AWS credentials by utilizing IAM Roles.
  • Team Ready: Entire monitoring stack can be updated via Git Pull Requests.
  • Zero Data Loss: Implemented Systemd Restart=always policies for all exporters.
